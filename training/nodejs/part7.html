7.1 The Backend Is Responsible for Security

A backend handles real user data and actions, making security a core, non-negotiable responsibility — one that distinguishes professional backend work from naive code. Two pillars matter most: authentication (proving who a user is — sign-up, login, knowing which user makes a request) and authorisation (controlling what they're allowed to do — e.g. only the owner can edit their data); plus general security (protecting against the common ways APIs get attacked or leak data). The golden rules 'never trust input' and 'secure your secrets' live here. Getting security wrong has serious consequences — leaked passwords, exposed data, account takeovers, the whole app compromised — so a backend developer must understand how to authenticate users safely, store credentials securely, validate input, and follow security basics. This competence separates a real backend developer from someone who can only build insecure toys, and it's heavily relevant to employability since companies care deeply about not getting breached.

7.2 Authentication and Security Basics (Practical)

AUTHENTICATION & SECURITY
  PASSWORDS (never store plain text!):
    * HASH passwords with bcrypt (or argon2) before storing: bcrypt.hash(password, 10)
    * on login, COMPARE: bcrypt.compare(input, storedHash)
  AUTHENTICATION (knowing who the user is across requests):
    * JWT (JSON Web Tokens): on login, issue a signed token; client sends it each request
      (Authorization: Bearer <token>); server verifies -> stateless, common for APIs
    * or SESSIONS (server-side session + cookie) -- also valid
    * AUTH MIDDLEWARE: checks the token & attaches the user (req.user), protecting routes
  AUTHORISATION: check the user is allowed (owns the resource / has the role)
  NEVER TRUST INPUT -- VALIDATE everything:
    * validate & sanitise request data (zod / Joi) -> reject bad input (400)
  SECURITY BASICS: secrets in ENV VARS (never in code/git) . HTTPS . helmet (headers) .
    CORS configured . rate limiting . keep deps updated (npm audit) . don't leak errors
KEY: hash passwords . JWT/session auth via middleware . VALIDATE all input . secrets in env . HTTPS.

Passwords: never store plain text — hash them with bcrypt (or argon2) before storing, and compare hashes on login. Authentication across requests: JWT (on login, issue a signed token the client sends each request in the Authorization header; the server verifies it — stateless, common for APIs) or server-side sessions; either way an auth middleware checks the token and attaches the user (req.user), protecting routes. Authorisation: check the user is actually allowed (owns the resource, has the role). Never trust input — validate and sanitise all request data (with zod or Joi), rejecting bad input with a 400. And security basics: secrets in env vars (never in code/git), HTTPS, secure headers (helmet), CORS, rate-limiting, updated dependencies (npm audit), and not leaking internal errors. These protect your API and its users.

7.3 Building Secure by Default

Professional backend developers build secure by default — treating security as fundamental to every feature, not bolted on later. This means: always hashing passwords (never even considering plain text), always validating and sanitising input at the boundary (assuming all external data is hostile until proven safe), always keeping secrets out of code, always protecting routes that need authentication, and always checking authorisation before letting a user act on data. It also means awareness of the common vulnerability classes (injection, broken authentication, exposed data — the kind catalogued in the OWASP Top 10) and basic defences, even without being a security specialist. This isn't paranoia — it's standard professional practice, because real APIs face real attacks and consequences, and 'I built an API but it wasn't secure' is a serious failure. Combining authentication, authorisation, input validation, and security basics turns your API into one that's safe to put real users and data behind. A secured, authenticated API — with hashed passwords, token auth, protected routes, and input validation — demonstrates the security competence essential for any real backend role and is a critical capstone milestone.

CHECKPOINT

Secure your API: add user sign-up/login with hashed passwords (bcrypt), token-based auth (JWT) via an auth middleware that protects routes and attaches the user, authorisation checks (users act only on their own data), and input validation (e.g. zod/Joi) that rejects bad data — with secrets in env vars. Building secure-by-default is essential professional backend practice and a critical capstone milestone.
