1.1 JavaScript on the Server

Node.js is a JavaScript runtime — a program that executes JavaScript outside the browser, on your computer or a server. It bundles Google's fast V8 engine (the same one in Chrome) with APIs for things browsers can't do: reading and writing files, making network connections, running servers, and accessing the operating system. So with Node, JavaScript can do everything a backend language needs: handle web requests, talk to databases, process files, run scheduled jobs, and build command-line tools. The conceptual leap: the JavaScript language is the same as in the browser, but the environment is completely different — no DOM, no window, no web page, but instead file access, networking, and server capabilities. Node is what lets one language span both front end (browser) and back end (server), which is its superpower and the reason for full-stack JavaScript. Understanding Node as 'JavaScript + server capabilities, running outside the browser' frames everything that follows.

1.2 The Event Loop and Non-Blocking I/O

The single most important concept that makes Node Node is its non-blocking, asynchronous, event-driven model, powered by the event loop. Most traditional servers handle each request with a thread that waits (blocks) for slow operations like database queries or file reads. Node instead runs on a single main thread that never waits: when it hits a slow I/O operation, it hands it off and immediately moves on, then handles the result via a callback when it's ready — the event loop continuously processing completed events. This is why Node excels at I/O-heavy, high-concurrency workloads (APIs, real-time apps) — it handles many simultaneous connections efficiently without a thread per request. The practical consequence, which dominates how you write Node, is that operations are asynchronous: you don't get a result immediately, you get it later (via async/await — Part 3). Understanding the event loop and non-blocking model explains why Node behaves as it does and why async is everywhere — foundational to writing correct Node code.

1.3 Running Node and Setting Up a Project (Practical)

NODE FUNDAMENTALS (hands-on)
  RUN JAVASCRIPT WITH NODE:
    node app.js          # run a JS file
    node                 # interactive REPL (try JS live)
  GLOBALS (browser has window; Node has these):
    * process    -- info about the running process (process.argv, process.env, process.exit)
    * __dirname / __filename -- current file/folder paths (CommonJS)
    * console, setTimeout, etc. (familiar from JS)
    * NO DOM, window, document (those are browser-only)
  A PROJECT:
    npm init -y          # create package.json (your project's manifest)
    npm install <pkg>    # add a dependency (Part 2)
    # add a script:  "scripts": { "start": "node app.js" }  ->  npm start
  ENVIRONMENT VARIABLES (config & secrets):
    process.env.MY_VAR   # read env vars; keep secrets OUT of code (a .env file + dotenv)
KEY: run JS with `node`, set up a project with package.json/npm, use process & env vars.

Get hands-on: run JavaScript with node app.js (or node for a REPL). Learn Node's globals — instead of the browser's window/document, Node gives you process (process.argv for command-line arguments, process.env for environment variables), __dirname/__filename for paths, plus familiar console and timers. Set up a project with npm init -y (creating package.json, your project's manifest with name, scripts, and dependencies), add dependencies with npm install, and define scripts (like "start": "node app.js"). And learn environment variables (process.env) for configuration and secrets — kept out of code, in a .env file. This setup underlies every Node project.

1.4 The Node Mental Model

Pulling Part 1 together gives the mental model that makes Node make sense. Node is JavaScript running on the server via V8 plus server APIs (files, networking, OS) — no browser, no DOM. It's asynchronous and event-driven, using a single-threaded event loop and non-blocking I/O to handle many operations efficiently without waiting — which is why your code is full of async/await. You run it with node, organise projects with package.json and npm, and configure it with environment variables. This model explains the rest of the guide: why you write async code, why you compose with modules and packages, why async patterns matter, how you access the system, and how you build servers. Grasping that Node is 'async JavaScript with server superpowers, organised by npm' is the foundation — and with a configured project running, you're ready to build. Many beginners struggle precisely because they miss the async/event-loop model; getting it now makes everything downstream click.
