11.1 What DevOps Means

Development is writing software; Operations is running it for real users. DevOps is the discipline (and toolset) that connects them: packaging your app so it runs anywhere, testing it automatically on every change, and deploying it to servers without drama. A junior developer who can deploy is worth two who cannot.

11.2 Environment Variables: The First Professional Habit

Secrets (database passwords, API keys) and per-environment settings (which port?) must never be written in code or committed to Git. They live in environment variables, supplied from outside the program:

11.5 Deploying for Real (Practical)

Static sites (HTML/CSS/JS, React builds) — Netlify

  1. Sign up at netlify.com with your GitHub account, click "Add new site" → "Import an existing project", and pick your repo.
  2. For a React (Vite) project set Build command: npm run build and Publish directory: dist. For plain HTML, no build command at all.
  3. Deploy. Every future git push redeploys automatically — that is CD working for you.

Backends (Node/Express APIs) — Render

  1. Sign up at render.com with GitHub, click New → Web Service, and pick your API repo.
  2. Build command: npm install. Start command: node server.js. Choose the free tier.
  3. Render sets the PORT environment variable — your 11.2 code already reads it, so it just works. Add any secrets in the Environment tab, never in code.
REALITY CHECK

Free-tier backends sleep when idle and take ~30 seconds to wake on the first request. That is normal and fine for portfolio projects — mention it in your README so reviewers expect it.

11.6 Checkpoint

  1. Your Part 6 React app is live on Netlify with a public URL.
  2. Your students API runs in Docker locally AND is deployed on Render.
  3. A GitHub Actions workflow runs your tests on every push, and you have seen it fail and fixed it.
CHECKPOINT

You can ship. From idea to a URL strangers can visit — the complete loop. Everything you build now goes live by default.
