14.1 Your Portfolio: Proof Beats Promises

Employers do not hire course-finishers; they hire people who ship. You have shipped. Now package it:

  1. Upgrade your Part 2 landing page into a portfolio site: short bio, skills, and 3–4 featured projects — MentorLink first, then the polyglot dashboard, the React catalogue, and one more you are proud of. Deploy on GitHub Pages or Netlify.
  2. Each project card links to BOTH the live demo and the GitHub repo. Reviewers click the live link first; broken links end interviews before they start. Test every link monthly.
  3. Every featured repo gets a real README: one-line pitch, screenshot, features, tech stack, how to run locally. A great README signals professionalism louder than the code.
  4. Pin your best repositories on your GitHub profile and add a profile README (a repo named exactly your username) introducing yourself in three sentences.

14.2 The Developer CV That Gets Replies

  • One page. Structure: contact + links (GitHub, portfolio, LinkedIn) → skills → projects → experience → education. As a career-switcher, projects go ABOVE experience.
  • Write impact, not duties: "Built and deployed a full-stack LMS (Node, SQLite, CI/CD on GitHub Actions) supporting registration, auth and progress tracking — live at <URL>" beats "learned web development".
  • Mirror the job advert's keywords (React, REST, SQL, Docker, Git) honestly — automated screeners match words before humans read sentences.
  • LinkedIn: headline "Full-Stack Developer | JavaScript · React · Node · Python · SQL", the same projects in Featured, and Open To Work switched on for recruiters.

14.3 Where the Jobs Are — Including Remote

  • Apply broadly in titles: Junior / Graduate Software Developer, Frontend Developer, Backend Developer, Full-Stack Developer, Software Engineer I, and technical support engineer roles that promote into development.
  • Remote-friendly boards: LinkedIn (filter Remote), Wellfound, RemoteOK, We Work Remotely — plus local tech hubs and communities, where referrals shortcut every queue.
  • Freelance early: one or two small real clients (a shop website, a booking form, an automation script) add paid, real-world lines to your CV while you interview.
  • The numbers game: expect 50–150 applications for a first role. It is a funnel, not a verdict. Ten tailored applications a week, every week.

14.4 Surviving the Technical Interview

  1. Practice 30–50 easy problems on a site like LeetCode or Codewars using JavaScript or Python — arrays, strings, objects/dicts, loops. Your Part 3 and Part 7 skills cover them.
  2. Narrate while you code: "First I will handle the empty case, then loop and track the maximum…". Interviewers grade your thinking more than your typing.
  3. Prepare a 2-minute walkthrough of MentorLink: the problem, the architecture (draw the diagram), one hard bug and how DIRE solved it, and what you would improve next. This story IS the interview.
  4. When stuck, say what you know, what you would look up, and how you would test a guess. "I don't know, but here is how I would find out" is a passing answer.

14.5 The Habits That Compound for the Next Ten Years

  • Ship something every week — a feature, a fix, a script. Your GitHub activity graph is your heartbeat.
  • Read code weekly: pick a small open-source repo and trace how it works. Reading is the fastest way to absorb professional patterns.
  • Learn in public: a short LinkedIn post per project — what you built, what broke, what you learned. Opportunities find visible people.
  • Go deep before wide: master this stack for 6–12 months before chasing new frameworks. Depth gets you hired; breadth comes free with the years.
THE FINISH LINE THAT ISN'T

Twenty-four weeks ago you could not open a terminal. Today you design schemas, build APIs, ship containers, and debug systematically. You are not "done learning" — no developer ever is — but you are, in every sense that matters, a software developer. Go act like one.

A.5 Free Resources Worth Your Time

  • MDN Web Docs (developer.mozilla.org) — the reference for HTML, CSS and JavaScript. Bookmark it; professionals check it daily.
  • Official docs: react.dev, docs.python.org, expressjs.com, docs.docker.com — always current, always correct.
  • Practice: Codewars and LeetCode (easy tier) for interview muscles; Frontend Mentor for design-to-code practice.
  • When stuck: read the error → DIRE → search the exact message → ask an AI assistant to explain (not write) the fix — in that order. Understanding compounds; pasting does not.
