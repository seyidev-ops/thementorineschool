8.1 From Components to a Real Application

A real app is more than components — it has multiple pages/views, a sensible structure, and it's deployed live for people to use; this part turns your React skills into a complete application. Routing: web apps have different pages or views (a home page, a details page, a profile, a dashboard), and in a React single-page app you handle this with client-side routing — showing different components for different URLs without full page reloads, giving a fast, app-like experience. The standard tool is React Router. Structure: as an app grows, organising it well — sensible folders (components, pages, hooks), reusable components, clear separation — keeps it maintainable. Deployment: getting your app built and live on the internet (not just running on your laptop) so it's a real, usable, shareable application — and importantly, deployed apps are what you show employers. Understanding routing, good structure, and deployment completes your ability to build and ship complete React applications, not just isolated components — which is what real work and a real portfolio require.

8.2 Routing, Structure, and Deploying (Practical)

ROUTING, STRUCTURE & DEPLOYMENT
  ROUTING with REACT ROUTER (multiple pages/views, no reloads):
    npm install react-router-dom
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/products" element={<Products />} />
        <Route path="/products/:id" element={<ProductDetail />} />   // URL param
        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
    <Link to="/products">Products</Link>     // navigate (no reload)
    useParams() (read :id) · useNavigate() (navigate in code)
  PROJECT STRUCTURE (keep it organised as it grows):
    src/ components/ (reusable UI) · pages/ (route views) · hooks/ (custom hooks) ·
        services/ or api/ (data fetching) · assets/ · App.jsx
    · one component per file · sensible names · reuse, don't repeat
  BUILD & DEPLOY:
    npm run build              # Vite builds an optimised production bundle (dist/)
    · deploy the build to: Netlify, Vercel, GitHub Pages, Cloudflare Pages (free tiers)
    · (these often auto-build & deploy straight from your GitHub repo — see the Git skill)
  ENV VARS for config/keys (Vite: import.meta.env.VITE_*) — never hard-code secrets
KEY: React Router for pages, organised structure, then build & deploy live (Netlify/Vercel/etc.).

Routing with React Router: wrap the app in <BrowserRouter>, define <Routes> mapping URL paths to components (including URL params like /products/:id and a catch-all ), and navigate with <Link to="..."> (no reload), reading params with useParams and navigating in code with useNavigate. Project structure: organise src/ into components/ (reusable UI), pages/ (route views), hooks/ (custom hooks), and services/ (data fetching) — one component per file, sensible names, reuse over repetition. Build and deploy: npm run build produces an optimised production bundle, which you deploy to Netlify, Vercel, GitHub Pages, or Cloudflare Pages (free tiers, often auto-building straight from your GitHub repo — see the Git and Cloudflare skills). Use env vars for config/keys (import.meta.env.VITE_), never hard-coding secrets. This turns your components into a live, multi-page app.

8.3 Shipping Complete React Apps

Routing, structure, and deployment are what let you ship complete React applications — the real deliverable — and they round out your professional competence. Routing makes your app feel like a real multi-page application (with proper URLs, navigation, and shareable links) while keeping the speed of a single-page app — essential for anything beyond a single screen. Good structure keeps your growing app maintainable and is something experienced developers and interviewers notice — a well-organised React project (clear folders, reusable components, custom hooks, separated concerns) signals competence, while a messy one signals the opposite. Deployment completes everything: a React app that's actually live — built, optimised, and hosted — is a real, usable, shareable product, and crucially it's what you put in your portfolio and show employers (a live link plus the GitHub repo is far more convincing than 'it runs on my machine'). The deployment workflow also connects to the broader ecosystem: hosting platforms (Netlify, Vercel, Cloudflare Pages) often deploy automatically from your GitHub repo on every push (tying into the Git skill's CI/CD), giving a professional, automated pipeline from commit to live app. (It's also worth knowing that in 2026 many production React apps use the Next.js framework, which adds routing, server rendering, and more on top of React — a strong next step once you know core React.) Building a multi-page, well-structured, deployed React app demonstrates you can ship complete, real applications — the milestone that makes your capstone a genuine, portfolio-ready product.

CHECKPOINT

Build a multi-page, deployed app: add client-side routing with React Router (several pages, a URL param, navigation links), organise the project sensibly (components/pages/hooks folders), then build it and deploy it live to a free host (Netlify/Vercel/Cloudflare Pages/GitHub Pages) — ideally auto-deploying from your GitHub repo. Routing + structure + a live deployment turn components into a complete, shippable React app — the milestone that completes your capstone.
