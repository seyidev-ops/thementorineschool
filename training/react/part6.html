6.1 Side Effects and the Outside World

Real apps need to interact with the outside world — most importantly, fetching data from APIs (loading the products, posts, or user data the app displays) — and in React this involves side effects: things that happen outside the normal render-from-state flow (data fetching, subscriptions, timers, manually interacting with the DOM). The hook for handling side effects is useEffect, which lets you run code after a component renders, and control when it runs (once on mount, or whenever certain values change). The most common and important use is data fetching: when a component loads (mounts), fetch data from an API, store it in state, and the component re-renders to display it — exactly how real apps populate themselves with live data. This connects React to the wider world (and to back ends like the Node APIs from the school's Node skill): the front end fetches from APIs, stores the data in state, and renders it. Understanding useEffect and the standard data-fetching pattern is essential for building real, dynamic apps that work with live data, which is what almost every real React app does.

6.2 useEffect and Fetching Data (Practical)

useEffect & DATA FETCHING
  useEffect (run code after render; control WHEN with the dependency array):
    import { useState, useEffect } from 'react';
    useEffect(() => {
      // side-effect code (e.g. fetch data)
    }, []);   // <- DEPENDENCY ARRAY: [] = run ONCE on mount; [x] = run when x changes;
              //    (omitted) = run after every render (rarely what you want)
  THE DATA-FETCHING PATTERN (the standard):
    function Users() {
      const [users, setUsers] = useState([]);
      const [loading, setLoading] = useState(true);
      const [error, setError] = useState(null);
      useEffect(() => {
        async function load() {
          try {
            const res = await fetch('https://api.example.com/users');
            if (!res.ok) throw new Error('Failed');
            setUsers(await res.json());          // store data in state -> re-render
          } catch (e) { setError(e.message); }
          finally { setLoading(false); }
        }
        load();
      }, []);   // fetch once on mount
      if (loading) return <Spinner />;            // handle the 3 states (Part 5)
      if (error) return <Error msg={error} />;
      return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
    }
  CLEANUP: return a function from useEffect to clean up (timers, subscriptions)
KEY: useEffect(fn, [deps]) runs effects; the fetch-on-mount + loading/error/data pattern is standard.

useEffect(fn, [deps]) runs code after render, with the dependency array controlling when: [] runs once on mount (the common case for initial data fetching), [x] runs whenever x changes, and omitting it runs after every render (rarely wanted). The standard data-fetching pattern: hold data, loading, and error in state; in a useEffect with [], run an async function that fetches, stores the result in state (setUsers(await res.json()) — triggering a re-render), catches errors into error state, and clears loading; then render the three states — spinner while loading, error message on failure, the mapped data on success (Part 5). useEffect can also return a cleanup function (for timers/subscriptions). This fetch-on-mount-with-loading/error/data pattern is how real React apps load their data.

6.3 Building Real, Data-Driven Apps

With useEffect and data fetching, you can build real, data-driven applications — the kind that work with live information and that make compelling portfolio pieces. Now your components can load data from any API (a public API, or your own back end) and display it: a list of posts, products, search results, weather, anything. This brings the whole React picture together: useEffect fetches the data, state stores it (and the loading/error status), conditional rendering (Part 5) shows the right UI for each state, list rendering (Part 5) maps the data to components, and events (Part 4) let users trigger fetches (like searching). The result is a genuine app that loads and displays real data — exactly what almost every React app does. It's also the front-end half of full-stack work: the React front end fetches from a back-end API (like a Node/Express API — the school's Node skill), forming a complete application. A few practical points matter: always handle the loading and error states (a good app doesn't just show blank while fetching or crash on failure — it shows feedback), fetch in useEffect with the right dependencies, and be mindful of fetching only when needed. (Note: in 2026, larger apps often use data-fetching libraries like TanStack Query, and frameworks like Next.js handle fetching differently — but understanding the core useEffect + fetch pattern is the essential foundation.) Building an app that fetches and displays live API data, handling loading and error states, demonstrates you can build real, dynamic React applications — a major capstone milestone.

CHECKPOINT

Build an app that fetches live data: use useEffect (with []) to fetch from a real API on mount, store data/loading/error in state, and render the three states — spinner, error message, and the mapped data — ideally letting the user trigger fetches (a search). Handle loading and error gracefully. The useEffect + fetch + loading/error/data pattern is how real React apps load data — a major milestone toward genuine, dynamic applications.
