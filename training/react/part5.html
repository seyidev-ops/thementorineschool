5.1 UI as a Function of Data

Most real UIs render collections of data (a list of products, posts, tasks, messages) and show different things in different situations (a loading spinner, an empty state, an error, content) — and React handles both elegantly because UI is a function of data. Rendering lists: you take an array of data and map it to an array of components — the same JavaScript array.map() you know, used directly in JSX — so a list of data becomes a list of UI elements, and when the data changes, the rendered list updates automatically. Conditional rendering: you use ordinary JavaScript logic (the ternary, &&, if-conditions) right in your JSX to decide what to show based on state or data — showing a spinner while loading, an error message on failure, the content on success, or different UI for logged-in vs logged-out users. Together, these are how React UIs become dynamic and data-driven — the UI always reflects the current data and state. Mastering list rendering and conditional rendering is essential because virtually every real app does both constantly, and they're where React's 'describe the UI for this data' model shines.

5.2 Mapping Lists and Conditional Rendering (Practical)

RENDERING LISTS & CONDITIONALS
  RENDER A LIST (map an array of data to components):
    {products.map(product => (
      <ProductCard key={product.id} name={product.name} price={product.price} />
    ))}
    · use array.map() (the JS method) right inside JSX, in { }
    · KEY: each item needs a unique, stable `key` prop (key={item.id}) — helps React
      track items efficiently (use a real id, NOT the array index if the list changes)
  CONDITIONAL RENDERING (show UI based on state/data — just JavaScript):
    {isLoading && <Spinner />}                      // && : show only if true
    {error ? <Error msg={error} /> : <Content />}   // ternary: either/or
    {items.length === 0 && <p>No items yet</p>}      // empty state
    {isLoggedIn ? <Dashboard /> : <Login />}
    (or use if-statements before the return to pick what to render)
  COMBINE: map a list, conditionally render an empty state, show loading/error (Part 6)
KEY: array.map(...) with a unique key renders lists; &&/ternary render UI conditionally.

Render a list by mapping an array of data to components right inside JSX (in { }): {products.map(product => <ProductCard key={product.id} ... />)} — using the same array.map() you know from JavaScript. Crucially, each item needs a unique, stable key prop (key={item.id}) so React can track items efficiently — use a real id, not the array index if the list can change. Conditional rendering uses ordinary JavaScript in JSX: && to show something only when true ({isLoading && <Spinner />}), the ternary for either/or ({error ? <Error /> : <Content />}), and conditions for empty states or logged-in/out UI. You combine them constantly — map a list, show an empty state when it's empty, and (with Part 6) show loading and error states. These are the core of dynamic, data-driven React UIs.

5.3 Dynamic, Data-Driven Interfaces

List rendering and conditional rendering are what make React UIs truly dynamic — the UI is a direct reflection of your data and state, automatically staying in sync. This is the payoff of React's declarative, state-driven model: you hold data in state, you map it to UI and conditionally render based on it, and whenever the state changes (an item added, data loaded, a filter applied), React re-renders and the UI updates to match — no manual DOM updates ever. This pattern — state/data → mapped and conditionally rendered → UI — is the essence of how React apps work and what you'll do constantly: rendering lists of fetched data, filtering and sorting them (using the JavaScript array methods on your data before mapping), showing the right UI for loading/empty/error/success states. The key prop is a small but important detail (it lets React efficiently update lists; getting it wrong causes subtle bugs), as is keeping rendering logic clean (complex conditionals can be extracted into variables or helper functions before the JSX). Mastering dynamic rendering ties together everything so far — components and props (the UI pieces), state (the changing data), events (user actions that change state) — into apps whose UI faithfully reflects their data. Building a dynamic, data-driven UI (a filterable list, a UI that shows different states) demonstrates you can build the data-reflecting interfaces that are the substance of real React apps, ready to power them with live API data.

CHECKPOINT

Build a dynamic, data-driven UI: render a list by mapping an array of data to components (with proper unique keys), and use conditional rendering (&&, ternary) to handle different states — an empty state, and ideally a loading/error/content split. Combine with state so the UI updates when data changes (e.g. a filterable or addable list). UI-as-a-function-of-data — mapped lists and conditional rendering — is the substance of every real React app.
