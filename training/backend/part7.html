7.1 Security Is a Feature, Not a Garnish

Everything you have built so far trusts everyone completely — anyone can read, edit or delete anything. Real systems must answer two questions on every request: Who are you? (authentication) and What are you allowed to do? (authorization). This part installs both, plus the defensive habits that keep you off the news.

7.2 Passwords: Hash, Never Store (Practical)

Storing a password is malpractice — databases leak, and leaked plaintext passwords destroy users who reuse them everywhere. Instead you store a hash: a one-way fingerprint computed by bcrypt, deliberately slow to compute so brute-force attacks become economically absurd. At login, you hash what the user typed and compare fingerprints.

FILEpart7-auth/hash-demo.js
import bcrypt from "bcrypt";

const password = "mentorine2026";
const hash = await bcrypt.hash(password, 10);   // 10 = cost factor
console.log("Stored in DB:", hash);

console.log("Right password:", await bcrypt.compare("mentorine2026", hash)); // true
console.log("Wrong password:", await bcrypt.compare("password123", hash));   // false

Run it twice — the hash differs each time (bcrypt salts automatically), yet compare still works. Magic, formally verified.

7.3 JWT: The Nightclub Wristband

HTTP has no memory — every request arrives as a stranger. A JSON Web Token fixes this like a club wristband: at login (showing ID once), the server signs a token containing who you are and hands it over; every later request presents the wristband in a header; the server verifies the signature — no database lookup needed — and trusts the contents. Tamper with the token and the signature breaks.

FILEpart7-auth/jwt-demo.js
import "dotenv/config";
import jwt from "jsonwebtoken";

// .env: JWT_SECRET=use-a-long-random-string-here
const token = jwt.sign(
  { userId: 42, name: "Ada" },
  process.env.JWT_SECRET,
  { expiresIn: "2h" }
);
console.log("Wristband:", token);

const decoded = jwt.verify(token, process.env.JWT_SECRET);
console.log("Verified wearer:", decoded);
COMMON MISTAKE

JWTs are signed, NOT encrypted — anyone can read the payload at jwt.io. Never put passwords or secrets inside one. The signature proves authenticity, not secrecy.

7.4 The Secure User System (Practical)

Now assemble register, login, and a protected route guarded by middleware — the airport security training from Part 4 paying off:

FILEpart7-auth/server.js
import "dotenv/config";
import express from "express";
import bcrypt from "bcrypt";
import jwt from "jsonwebtoken";

const app = express();
app.use(express.json());
const users = [];   // swap for SQLite/Mongo using Parts 5–6 — upgrade challenge

app.post("/auth/register", async (req, res) => {
  const { email, password } = req.body;
  if (!email || !password || password.length < 8)
    return res.status(400).json({ error: "Valid email and 8+ char password required" });
  if (users.find(u => u.email === email))
    return res.status(409).json({ error: "Email already registered" });
  users.push({ id: users.length + 1, email, passwordHash: await bcrypt.hash(password, 10) });
  res.status(201).json({ message: "Registered. Please log in." });
});

app.post("/auth/login", async (req, res) => {
  const user = users.find(u => u.email === req.body.email);
  const ok = user && await bcrypt.compare(req.body.password || "", user.passwordHash);
  if (!ok) return res.status(401).json({ error: "Invalid credentials" });   // never say which part failed
  const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET, { expiresIn: "2h" });
  res.json({ token });
});

// The guard: verifies the wristband, attaches the wearer to req
const requireAuth = (req, res, next) => {
  const header = req.headers.authorization || "";
  const token = header.startsWith("Bearer ") ? header.slice(7) : null;
  if (!token) return res.status(401).json({ error: "Token required" });
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch {
    return res.status(401).json({ error: "Invalid or expired token" });
  }
};

app.get("/me/tasks", requireAuth, (req, res) => {
  res.json({ message: `Private tasks for user #${req.user.userId}` });
});

app.listen(4000, () => console.log("Auth server on :4000"));

Walk the whole journey with curl — register, log in, copy the token, and present the wristband:

curl -X POST http://localhost:4000/auth/register -H "Content-Type: application/json" -d '{"email":"ada@x.com","password":"longpassword"}'
curl -X POST http://localhost:4000/auth/login -H "Content-Type: application/json" -d '{"email":"ada@x.com","password":"longpassword"}'
curl http://localhost:4000/me/tasks -H "Authorization: Bearer PASTE_TOKEN_HERE"
curl -i http://localhost:4000/me/tasks    # no wristband -> 401

7.5 The OWASP Hall of Shame: Mistakes You Will Never Make

Leaked secretsKeys committed to GitHubEverything secret lives in .env — Part 3
Plaintext passwordsStoring what users typedbcrypt hash + compare — Part 7
Information leakageStack traces sent to clientsCentral error handler, log privately — Part 4
Broken access controlTrusting the client's claimsrequireAuth middleware + checking ownership server-side

Two more guards every public API wears: CORS (npm install cors — declares which websites may call you) and rate limiting (npm install express-rate-limit — caps requests per visitor so brute-forcers and accidental loops can't drown you). Both are two-line middleware; add them to the checkpoint.

7.6 Checkpoint Project: The Secure Task API

Merge everything: take your persistent Task API from Part 5, add the auth system, give the tasks table a user_id column, and make every task route both requireAuth-protected and ownership-checked — users see and touch only their own rows (WHERE user_id = ?). Add CORS and a rate limiter on /auth/login (try 5 attempts per minute). When two different registered users each maintain private task lists on the same server, you have built what most tutorials never reach: a genuinely multi-user, secured back end.
