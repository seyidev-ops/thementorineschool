4.1 A Web Server in Eleven Lines

Express is the most-used web framework in the Node world — a thin, elegant layer that turns “answer HTTP requests” into a few readable lines. Set up a fresh project and meet your first server:

cd backend-journey/part4-express
npm init -y
npm install express
# add "type": "module" to package.json as before
FILEpart4-express/server.js
import express from "express";

const app = express();
app.use(express.json());          // understand JSON request bodies

app.get("/", (req, res) => {
  res.json({ message: "The Mentorine Task API is alive" });
});

const PORT = 4000;
app.listen(PORT, () => {
  console.log(`Server running at http://localhost:${PORT}`);
});

Run node server.js, open a second terminal (the first is now busy being a server), and:

curl http://localhost:4000/

You just answered your first HTTP request with your own code. The req object is everything the client sent; res is your microphone back to them. Stop the server anytime with Ctrl+C.

PRO HABIT

Run node --watch server.js instead and Node restarts automatically every time you save — the standard development rhythm.

4.2 Routing: Different Paths, Different Answers

A route is verb + path + handler. Path parameters (:id) capture values from the URL; query strings (?done=true) carry options. Add these below your first route:

FILEpart4-express/server.js (add routes)
const tasks = [
  { id: 1, title: "Finish Part 4", done: false },
  { id: 2, title: "Drink water", done: true }
];

// GET all tasks, with optional filtering: /tasks?done=true
app.get("/tasks", (req, res) => {
  if (req.query.done !== undefined) {
    const want = req.query.done === "true";
    return res.json(tasks.filter(t => t.done === want));
  }
  res.json(tasks);
});

// GET one task by id: /tasks/1
app.get("/tasks/:id", (req, res) => {
  const task = tasks.find(t => t.id === Number(req.params.id));
  if (!task) return res.status(404).json({ error: "Task not found" });
  res.json(task);
});
TRY THIS

curl http://localhost:4000/tasks?done=true — then ask for /tasks/99 and confirm you get a clean 404 with a JSON error, not a crash. Honest errors are a feature.

4.3 Middleware: The Airport Security Line

Middleware is a chain of checkpoints every request walks through before reaching its gate. Each checkpoint can inspect the request, stamp it, reject it, or wave it on with next(). Logging, JSON parsing, authentication (Part 7), rate limiting — all middleware. Write your first:

FILEpart4-express/server.js (above the routes)
// Logger: every request gets a line in the terminal
app.use((req, res, next) => {
  console.log(`${new Date().toISOString()}  ${req.method} ${req.url}`);
  next();   // forgetting next() leaves the request hanging forever
});

Order matters: middleware runs top to bottom, so the logger must be registered before the routes it should observe. You already used built-in middleware without knowing — express.json() is the checkpoint that unpacks JSON bodies into req.body.

4.4 Full CRUD: The Task Manager API (Practical)

CRUD — Create, Read, Update, Delete — is the heartbeat of practically every product. You have Read; now complete the set:

FILEpart4-express/server.js (add below the GET routes)
// CREATE
app.post("/tasks", (req, res) => {
  const { title } = req.body;
  if (!title || !title.trim()) {
    return res.status(400).json({ error: "title is required" });
  }
  const task = { id: Date.now(), title: title.trim(), done: false };
  tasks.push(task);
  res.status(201).json(task);          // 201 = Created
});

// UPDATE
app.put("/tasks/:id", (req, res) => {
  const task = tasks.find(t => t.id === Number(req.params.id));
  if (!task) return res.status(404).json({ error: "Task not found" });
  if (req.body.title !== undefined) task.title = req.body.title;
  if (req.body.done !== undefined) task.done = Boolean(req.body.done);
  res.json(task);
});

// DELETE
app.delete("/tasks/:id", (req, res) => {
  const index = tasks.findIndex(t => t.id === Number(req.params.id));
  if (index === -1) return res.status(404).json({ error: "Task not found" });
  tasks.splice(index, 1);
  res.status(204).end();               // 204 = success, nothing to say
});

Exercise the whole lifecycle from your second terminal:

curl -X POST http://localhost:4000/tasks -H "Content-Type: application/json" -d '{"title":"Master CRUD"}'
curl http://localhost:4000/tasks
curl -X PUT http://localhost:4000/tasks/1 -H "Content-Type: application/json" -d '{"done":true}'
curl -i -X DELETE http://localhost:4000/tasks/2

4.5 Validation and Error Handling Like a Professional

Clients send nonsense — not maliciously (usually), but constantly. A professional API validates everything at the door and answers mistakes with helpful 400s. Centralise disaster handling with Express's special four-argument middleware at the very bottom of the file:

FILEpart4-express/server.js (at the very bottom, before listen)
// Any route nobody claimed -> clean 404
app.use((req, res) => {
  res.status(404).json({ error: `No route for ${req.method} ${req.url}` });
});

// Any uncaught explosion -> clean 500, details stay in YOUR logs
app.use((err, req, res, next) => {
  console.error(err);                       // full story for you
  res.status(500).json({ error: "Something went wrong on our side" });   // calm message for them
});
GOLDEN RULE

Never leak stack traces or database errors to clients — they reveal your architecture to attackers. Log the truth privately; respond with dignity publicly.
