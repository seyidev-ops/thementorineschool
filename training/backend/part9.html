9.1 Automated Testing: Your Robot Quality Inspector

Professionals do not re-curl every endpoint after every change — they write tests once and let the machine re-check forever. Node ships a test runner in the box (node:test), and the pattern is simply: arrange, act, assert.

FILEpart9-capstone/tests/fees.test.js
import { test } from "node:test";
import assert from "node:assert";
import { calculateFee } from "../fees.js";

test("fee is 1.5% of small amounts", () => {
  assert.strictEqual(calculateFee(100000), 1500);
});

test("fee is capped at 2000", () => {
  assert.strictEqual(calculateFee(10000000), 2000);
});

test("fee on zero is zero", () => {
  assert.strictEqual(calculateFee(0), 0);
});
node --test

Green ticks scroll by; break the cap in fees.js on purpose and watch a test catch you in milliseconds. For testing whole endpoints, the supertest package fires requests at your Express app in-memory — add it to the capstone as an upgrade. The professional rule of thumb: every bug you fix earns a test that would have caught it.

9.2 Git & GitHub: The Professional Workflow

Git records snapshots of your project; GitHub publishes them. The daily loop is four commands, and from today every checkpoint you have built goes into it:

cd backend-journey
git init
git add .
git commit -m "Parts 1-8: terminal to payments"
# create an empty repo on github.com, then:
git remote add origin https://github.com/YOUR-USERNAME/backend-journey.git
git push -u origin main

Commit small and often, with messages that say why: git commit -m "Reject negative deposits (caught by Ada in review)" is documentation your future self will thank you for. Your .gitignore from Part 3 keeps node_modules/ and .env out — verify on GitHub that neither appears. A recruiter checking your profile and finding a steady green commit graph plus documented projects is the cheapest interview advantage that exists.

9.3 Docker in Twenty Minutes

“Works on my machine” dies here. Docker packages your app with its environment — Node version, dependencies, OS libraries — into an image that runs identically everywhere. One file describes the recipe:

FILEpart9-capstone/Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 4000
CMD ["node", "server.js"]
docker build -t naijastore .
docker run -p 4000:4000 --env-file .env naijastore

Read the Dockerfile top to bottom: start from a slim official Node image, copy the dependency manifests first (so Docker caches the slow npm ci layer), copy the code, declare the port, state the start command. That mental model — layers, cached top-down — is most of practical Docker.

9.4 CI/CD: Robots That Test Every Push

Continuous Integration means GitHub runs your tests automatically on every push — a tireless reviewer who never lets broken code sneak in. One small file in your repo switches it on:

FILE.github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: node --test

Push it, open the Actions tab on GitHub, and watch a fresh Ubuntu machine boot, install your project and run your tests — in about a minute, for free. The green badge it earns belongs at the top of your README.

9.5 Deployment: Your API on the Real Internet (Practical)

Render (render.com) deploys straight from GitHub on a free tier. The steps: push your project → New ▸ Web Service → connect the repo → build command npm ci, start command node server.js → add your environment variables (JWT_SECRET, PAYSTACK_SECRET, database URL) in the dashboard — the production .env → deploy. Two code details make any Node app deploy-ready:

FILEserver.js (deploy-ready details)
// 1. The platform chooses your port — accept it
const PORT = process.env.PORT || 4000;
app.listen(PORT, () => console.log(`Live on :${PORT}`));

// 2. A health check route platforms (and you) can ping
app.get("/health", (req, res) => res.json({ status: "ok", uptime: process.uptime() }));

Minutes later you hold a URL like https://naijastore-api.onrender.com — and the curl commands you have typed at localhost all week now work from any computer on Earth. Send one to a friend. Savour it. (For the database in production, Render and Railway both offer free Postgres; swapping SQLite for Postgres is a connection-string change with the pg package — the SQL you wrote is the same.)

PRO HABIT

Add console.log lines with context ("order 123 paid, ref xyz") — on Render they stream into the Logs tab, your eyes in production. Structured logging tools come later; the habit starts now.

Build order (one to two weeks, exactly as a professional would sequence it): (1) schema on paper — users, products, cart_items, orders, order_items, with foreign keys drawn as arrows; (2) auth, tested; (3) product CRUD with an is_admin column gating writes; (4) cart and orders inside a SQL transaction so stock and order rows change together or not at all; (5) payments and the webhook; (6) receipts and analytics; (7) tests for the money paths at minimum; (8) Docker, CI, deploy. Commit at every step.

CHECKPOINT

Done means: a stranger with only your README and a curl client can register, browse, fill a cart, pay with the test card, and receive a receipt — against your live Render URL. That stranger will eventually be an interviewer.

9.7 The Job Hunt: Turning the Capstone into Offers

Your portfolio is now arguments, not claims: a live API URL, a documented GitHub repo with green CI, a commit history showing sixteen weeks of steady work. Package it: a README with an architecture diagram (even hand-drawn and photographed), a Postman collection export so reviewers can click through your endpoints, and a five-line “what I'd build next” section showing product thinking. Target roles titled Back-End Developer (Node), API Developer, or Junior Software Engineer; in interviews, walk through NaijaStore's payment flow on a whiteboard — webhook signature verification included — and you will be discussing engineering as a peer, not auditioning as a student.
