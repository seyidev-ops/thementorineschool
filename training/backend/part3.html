3.1 What Node.js Actually Is

Node.js is Chrome's JavaScript engine (V8) lifted out of the browser and given new powers: reading files, opening network connections, talking to the operating system. The browser deliberately forbids those things for safety; the server requires them. Same language, new permissions.

One mental model carries you far: Node is single-threaded but never waits. One chef, many dishes, nothing blocking. That is why your async training in Part 2 was not optional — it is Node's whole personality.

3.2 Modules: Splitting Code Across Files

Real projects are many small files, each exporting what others may use. Node's modern module system uses export and import. To enable it, your project needs a package.json with "type": "module" — you will create one in the next section; for now, see the shape:

FILEpart3-node/fees.js
export const calculateFee = (amount) => Math.min(amount * 0.015, 2000);
export const naira = (n) => `\u20a6${n.toLocaleString()}`;
FILEpart3-node/app.js
import { calculateFee, naira } from "./fees.js";

console.log(`Fee on ${naira(80000)}: ${naira(calculateFee(80000))}`);

./fees.js — the dot-slash means “in this same folder”. Forgetting it is the single most common import error in Node.

3.3 npm: The World's Largest Code Library

npm is two things: a command-line tool and a registry of over two million free packages. Professional back-end work is largely assembling well-tested packages and writing the logic that is unique to your product. Initialise a project and install your first package:

cd backend-journey/part3-node
npm init -y
npm install chalk

npm init -y creates package.json — your project's identity card, listing its name and dependencies. Open it and add one line so modern imports work: "type": "module", right under the name field. The install command created node_modules/ (the actual library code — never edit, never upload to GitHub) and package-lock.json (exact version receipts).

FILEpart3-node/colour-demo.js
import chalk from "chalk";

console.log(chalk.green("\u2713 Payment processed"));
console.log(chalk.yellow("\u26a0 Low balance warning"));
console.log(chalk.red.bold("\u2717 Transaction failed"));
PRO HABIT

Before writing anything generic — date formatting, validation, IDs — search npm first. Someone has solved it, tested it, and documented it. Your job is judgement, not reinvention.

3.4 The File System: Node's Superpower

The fs/promises module reads and writes real files — config, logs, uploads, reports. Combined with JSON.stringify and JSON.parse, a plain text file becomes a tiny database (your training wheels until Part 5).

FILEpart3-node/files-demo.js
import { writeFile, readFile } from "fs/promises";

const students = [
  { name: "Ada", course: "backend" },
  { name: "Femi", course: "frontend" }
];

const main = async () => {
  // WRITE: object -> JSON text -> disk
  await writeFile("students.json", JSON.stringify(students, null, 2));
  console.log("Saved students.json");

  // READ: disk -> JSON text -> object
  const raw = await readFile("students.json", "utf-8");
  const loaded = JSON.parse(raw);
  console.log(`Loaded ${loaded.length} students. First: ${loaded[0].name}`);
};

main();
TRY THIS

Open students.json in VS Code — there is your data, human-readable. Add a third student by hand, run the script again, and watch your code pick up the change.

3.5 Environment Variables: Secrets Stay Out of Code

Database passwords, API keys, payment secrets — these must never be typed into source code, because source code gets shared, uploaded and leaked. Instead they live in the environment: a .env file on each machine, read at startup, and excluded from Git.

FILEpart3-node/.env
PORT=4000
DB_PASSWORD=super-secret-do-not-commit
PAYSTACK_KEY=sk_test_xxxxxxxxxxxxx
FILEpart3-node/env-demo.js
import "dotenv/config";   // npm install dotenv

console.log(`Server will run on port ${process.env.PORT}`);
console.log(`Key loaded: ${process.env.PAYSTACK_KEY ? "yes" : "NO - check .env"}`);

Create a .gitignore file containing two lines — node_modules/ and .env — and this discipline becomes automatic. Every deployment platform (Part 9) has a settings screen where you set the same variables in production.

WARNING

A leaked payment key is money gone. The .env pattern is not style — it is the first security habit of your career. Part 7 builds on it.

3.6 Checkpoint Project: The Notes CLI

Build a command-line notes tool: node notes.js add "Buy data bundle" appends a note; node notes.js list shows all notes; node notes.js delete 2 removes one. Notes persist in notes.json between runs — your first program with memory.

FILEpart3-node/notes.js
import { readFile, writeFile } from "fs/promises";

const FILE = "notes.json";

const load = async () => {
  try { return JSON.parse(await readFile(FILE, "utf-8")); }
  catch { return []; }                  // first run: no file yet
};
const save = (notes) => writeFile(FILE, JSON.stringify(notes, null, 2));

const [,, command, ...rest] = process.argv;   // words typed after node notes.js
const notes = await load();

if (command === "add") {
  notes.push({ text: rest.join(" "), created: new Date().toISOString() });
  await save(notes);
  console.log(`Added. You now have ${notes.length} note(s).`);
} else if (command === "list") {
  notes.forEach((n, i) => console.log(`${i + 1}. ${n.text}`));
  if (!notes.length) console.log("No notes yet.");
} else if (command === "delete") {
  const index = Number(rest[0]) - 1;
  if (!notes[index]) { console.log("No such note."); }
  else {
    const [removed] = notes.splice(index, 1);
    await save(notes);
    console.log(`Deleted: "${removed.text}"`);
  }
} else {
  console.log("Usage: node notes.js add <text> | list | delete <number>");
}

Upgrade challenges: add a clear command; colour the output with chalk; add a search <word> command using filter. When this works, notice what you have actually built: input parsing, persistence, validation, CRUD. That is an API without the network — and the network arrives next.
