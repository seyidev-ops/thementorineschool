6.1 Reacting to the User

Interactivity means responding to what the user does — clicks, typing, submitting forms, hovering, key presses — and JavaScript handles this through events. An event is something that happens (a button is clicked, an input changes, a form is submitted), and you write event handlers — functions that run in response to those events. This is what makes a page truly interactive rather than static: 'when this button is clicked, do this', 'when this form is submitted, validate and process it', 'when this input changes, update the display'. Combined with the DOM (Part 5), events complete the front-end loop: the user does something → an event fires → your handler runs → it updates the data and the DOM → the user sees the result. Forms are an especially important case — they're how users input data (sign-ups, searches, settings, any data entry), so handling form submission, capturing input values, and validating them is a core, constant front-end task. Mastering events and forms is what lets you build interfaces that actually do things in response to users — i.e. real apps.

6.2 Event Handling and Forms (Practical)

EVENTS & FORMS
  EVENT LISTENERS (run a function when something happens):
    button.addEventListener('click', () => { ... });
    input.addEventListener('input', (e) => { console.log(e.target.value); });
    common events: click, input, change, submit, keydown, mouseover, ...
    the EVENT OBJECT (e): info about the event — e.target (the element), e.target.value, etc.
  FORMS (capturing user input):
    form.addEventListener('submit', (e) => {
      e.preventDefault();              // STOP the default page reload (important!)
      const name = nameInput.value;    // read input values
      // ...validate, then process the data...
    });
  VALIDATION (never trust input — check it):
    · check required fields, formats (email), lengths, etc.
    · show clear error messages (update the DOM) · give feedback
  THE FULL LOOP: user acts -> event fires -> handler runs -> update data -> update DOM (Part 5)
  EVENT DELEGATION: one listener on a parent for many children (efficient for dynamic lists)
KEY: addEventListener to react; for forms, preventDefault + read .value + validate + give feedback.

Event listeners: element.addEventListener('click', handler) runs a function when an event fires — common events being click, input, change, submit, keydown, mouseover — and the handler receives an event object (e) with details like e.target (the element) and e.target.value (an input's value). Forms: listen for the submit event, call e.preventDefault() to stop the default page reload (important!), read the input .values, then validate and process them. Validation ('never trust input'): check required fields, formats (email), and lengths, showing clear error messages by updating the DOM. This completes the full front-end loop: user acts → event fires → handler runs → data updates → DOM updates (Part 5). And event delegation (one listener on a parent for many children) handles dynamic lists efficiently.

6.3 Building Real Interactive Apps

Events and forms, combined with everything so far, let you build real interactive applications — the kind of projects that prove you can do front-end work and that make great portfolio pieces. Think of the classic learning apps that are genuinely impressive: a to-do list (add tasks via a form, mark them done, delete them, render the list from data), a calculator, a quiz, a weather app (Part 7 adds the live data), an expense tracker, an interactive form with validation. Each brings the whole picture together: data structures (Part 3) hold the state, the DOM (Part 5) renders it, events (this part) respond to the user, and functions (Part 2) organise the logic — the complete front-end loop in action. Building these teaches you to structure an interactive app: where the data lives, how the user's actions change it, and how the display updates to match — which is exactly the mental model that React formalises. It also surfaces real concerns: validating and handling user input gracefully, giving clear feedback, and keeping the code organised as the app grows. A working interactive app — responding to user events, handling form input with validation, and updating the page — is a satisfying, portfolio-worthy demonstration that you can build the interactive front ends that are the most common entry point into web development careers.

CHECKPOINT

A real interactive app is a portfolio-worthy proof you can build the front ends that launch web-development careers.
