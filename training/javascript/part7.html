7.1 The Web Is Asynchronous

Real web apps don't work with only local data — they fetch data from servers and APIs (a weather app gets live weather, a shop loads products, a social app loads posts), and this happens asynchronously: you request the data, and it arrives later (network requests take time). Understanding and handling async JavaScript is therefore essential for building real apps, and it's the concept beginners most often stumble on (the golden rule 'embrace async'). The key idea: when you ask for data over the network, your code can't just wait and freeze — instead it continues, and you handle the data when it arrives. JavaScript's modern tools for this are promises (an object representing a value that will be available later) and async/await (syntax that makes asynchronous code read almost like normal sequential code). Combined with fetch (the built-in way to request data from APIs) and JSON (the format that data comes in — which you already understand from Part 3), you can connect your apps to the vast world of web APIs and live data. Mastering async and API calls is what turns static practice apps into real, dynamic applications.

7.2 Promises, Async/Await, and Fetch (Practical)

ASYNC JAVASCRIPT & APIs
  THE PROBLEM: network requests take time -> the result comes LATER (asynchronously)
  PROMISES: an object representing a future value (resolves with data, or rejects with error)
  ASYNC / AWAIT (the modern way — reads like normal code):
    async function loadUsers() {
      try {
        const response = await fetch('https://api.example.com/users');  // wait for response
        if (!response.ok) throw new Error('Request failed: ' + response.status);
        const users = await response.json();   // parse the JSON body (also async)
        return users;                          // use the data (e.g. render to the DOM)
      } catch (error) {
        console.error('Failed to load:', error);   // handle errors!
        // show the user a friendly error message (update the DOM)
      }
    }
  FETCH: the built-in way to call APIs (GET data, or POST/PUT/DELETE with options)
    fetch(url, { method:'POST', headers:{'Content-Type':'application/json'},
                 body: JSON.stringify(data) })
  THE FLOW: fetch -> await response -> response.json() -> use the data (render it)
  HANDLE STATES: loading (show a spinner) · success (render) · error (friendly message)
KEY: async/await + fetch + try/catch + .json() = get live data from APIs into your app.

Promises represent a future value (resolving with data or rejecting with an error). Async/await is the modern way to handle them: mark a function async, await fetch(url) to get the response, check response.ok, await response.json() to parse the JSON body, then use the data — all wrapped in try/catch for error handling. fetch is the built-in way to call APIs (GET data by default, or POST/PUT/DELETE with options including headers and a JSON body). The typical flow: fetch → await the response → .json() → use the data (render it to the DOM). And handle the states a real app has: loading (show a spinner), success (render the data), and error (show a friendly message). This is how your apps get live data from the wider world.

7.3 Building Real, Dynamic Apps

Fetching data from APIs is what lets you build real, dynamic applications — the kind that work with live, real-world information — and it brings your whole skill set together. Now you can build: a weather app (fetch live weather for a city), a movie or book search app (query a public API), a currency converter, a GitHub profile viewer, a news reader, a data dashboard — apps that pull real data and display it interactively. Each combines everything: fetch and async/await (this part) get the data, JSON and array methods (Part 3) process it, the DOM (Part 5) renders it, events and forms (Part 6) let the user request it (e.g. searching), and good async error handling (Part 8) makes it robust. This is also a crucial professional skill because almost every real app talks to APIs — front ends fetch from back ends (like the Node APIs from the school's Node skill), apps integrate third-party services, and data flows over the network constantly. Handling that data flow well — fetching, awaiting, parsing, rendering, and gracefully handling loading and error states — is core front-end competence. It's also exactly the pattern React apps use (fetch data, store it in state, render it). Building apps that fetch and display live data from real APIs is a major milestone: it proves you can build genuine, dynamic, real-world applications, not just static exercises — a compelling portfolio centrepiece.

CHECKPOINT

Build apps that fetch live data: use async/await and fetch to call a real public API, parse the JSON, and render it to the page — handling loading and error states gracefully — e.g. a weather app, a search app querying a public API, or a data viewer. Combine it with events/forms so the user drives it. Fetching and displaying live API data proves you can build real, dynamic applications — a compelling portfolio centrepiece and exactly the pattern React uses.
