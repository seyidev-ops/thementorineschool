8.1 Errors and Debugging Are the Job

A huge part of real programming is dealing with things going wrong — handling errors gracefully and debugging when code misbehaves — and getting good at this is what separates a frustrated beginner from a capable developer. Code will have bugs and hit errors (bad input, failed network requests, unexpected states); the question is whether your code handles them gracefully and whether you can find and fix problems efficiently. Error handling means anticipating what can go wrong (a fetch fails, a value is missing, input is invalid) and dealing with it cleanly (try/catch around risky operations, checking for missing data, showing users friendly messages instead of crashing). Debugging is the systematic skill of finding why code isn't working — reading error messages, using console.log and the browser's debugger to inspect what's happening, and narrowing down the problem. These are everyday realities of programming, and being confident with them — rather than panicking at a red error — makes you far more effective and is exactly what professional work requires. The mindset shift: errors and bugs aren't failures, they're normal parts of programming that you have the tools to handle.

8.2 Handling Errors and Debugging (Practical)

ERROR HANDLING, DEBUGGING & CLEAN CODE
  ERROR HANDLING:
    try { riskyOperation(); }
    catch (error) { console.error(error); /* show friendly message, handle gracefully */ }
    · wrap risky things (fetch, parsing, anything that can fail) in try/catch
    · check for missing/invalid data BEFORE using it (guard clauses, optional chaining)
    · throw your own errors when appropriate: throw new Error('message')
  DEBUGGING (find why it's broken — systematically):
    · READ the error message: type, message, and the line/stack — it usually tells you a lot
    · console.log values to see what's actually happening (your #1 tool)
    · use the browser DEBUGGER: breakpoints, step through code, inspect variables (DevTools > Sources)
    · narrow it down: isolate where it breaks; check assumptions one by one
  CLEAN CODE (read far more than written):
    · clear, descriptive NAMES · small, single-purpose FUNCTIONS · consistent formatting
    · avoid repetition (DRY) · comment the WHY (not the obvious) · organise into modules
KEY: try/catch + guard against bad data · read errors + console.log + debugger · clean, named, modular code.

Error handling: wrap risky operations (fetch, parsing, anything that can fail) in try/catch, handling errors gracefully (logging them, showing friendly messages rather than crashing); check for missing or invalid data before using it (guard clauses, optional chaining); and throw your own errors when appropriate. Debugging systematically: read the error message (type, message, line/stack — it tells you a lot), use console.log to see what's actually happening (your #1 tool), use the browser debugger (breakpoints, stepping through code, inspecting variables in DevTools), and narrow down where it breaks by checking assumptions one at a time. Clean code: clear descriptive names, small single-purpose functions, consistent formatting, avoiding repetition (DRY), commenting the why, and organising into modules. These are the everyday craft of real development.

8.3 Writing Code Like a Professional

Beyond fixing problems, this part is about writing code like a professional — robust and clean — which is what makes you employable and your projects credible. Robustness means your code handles the real world: it anticipates failures (a fetch might fail, data might be missing), handles them gracefully, validates input, and doesn't crash or behave bizarrely when something unexpected happens. Clean code means your code is readable and maintainable: clear names that explain themselves, functions that each do one thing, sensible organisation (splitting code into modules), no needless repetition, and just enough commenting to explain the non-obvious. This matters enormously because code is read far more than it's written — by teammates, by reviewers, by your future self — and clean code is faster to understand, debug, and extend, while messy code becomes a liability. Professional developers are judged not just on whether their code works, but on whether it's robust and clean, and these qualities are exactly what reviewers notice in your portfolio projects (and in interviews). The combination — anticipating and handling errors, debugging confidently, and writing clean, organised, readable code — is what turns 'I can make it work' into 'I write quality software', which is the real bar for professional work. Polishing your projects to be robust and clean demonstrates this professional maturity, making your whole portfolio more convincing.

CHECKPOINT

Make your projects robust and clean: add proper error handling (try/catch around risky operations, guard against missing data, friendly error messages), debug confidently (read errors, use console.log and the browser debugger), and refactor for clean code (clear names, small functions, modules, no repetition). Robust, clean code — not just code that works — is the real professional bar and what reviewers notice in your portfolio and interviews.
