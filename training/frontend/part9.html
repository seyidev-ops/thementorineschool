9.1 Debugging Like a Professional

The Five Errors Every Beginner Meets

Error MessageWhat It Actually MeansFirst Thing to Check
Uncaught TypeError: Cannot read properties of undefinedYou used something that doesn't exist yetSpelling; whether the data has loaded before you used it
Uncaught ReferenceError: x is not definedVariable name typo or not declaredSpelling and the order of your <script> tags
Uncaught SyntaxError: Unexpected tokenBroken punctuationMissing bracket, quote, or comma near the line number shown
404 Not Found (Network tab)Wrong file path or URLThe exact path — case-sensitive on real servers!
CORS policy errorServer hasn't allowed your originAdd cors() middleware to your API (Part 8.2)

The debugging method: read the FIRST line of the error, note the file and line number, console.log the suspicious values just before the crash line, and form a hypothesis before changing anything. Random changes are not debugging.

DevTools mastery: Elements tab to inspect live HTML/CSS, Console for errors and logs, Network tab to see every request (status, payload, response), Sources tab to set breakpoints that pause your code mid-run.

9.2 Performance & Accessibility: The Professional's Edge

Open DevTools → Lighthouse tab → Analyze page load. Lighthouse scores your site on Performance, Accessibility, Best Practices, and SEO. Run it on every project and chase 90+ on all four. The fixes that matter most:

Compress images — a 4MB photo is the #1 cause of slow sites. Use https://squoosh.app, export as WebP, and set width/height attributes.

Alt text on every image, label on every input — you already learned this in Part 1; Lighthouse verifies you actually did it.

Colour contrast — light gray text on white fails. Check pairs at https://webaim.org/resources/contrastchecker.

Semantic landmarks — <header>, <main>, <footer>, one <h1>, headings in order. Free points, and the right thing to do.

9.3 Deploying: Putting Your Work on the Real Internet

Static Sites & Vite/React Apps — Netlify or Vercel, free

Push your project to GitHub (you have been doing this daily since Part 6).

Sign up at netlify.com or vercel.com with your GitHub account and click “Import project”.

For Vite projects the defaults are detected automatically (build command npm run build, output dist). Click Deploy.

Ninety seconds later your app is live on a real URL — and every future git push redeploys automatically. This is professional CI/CD, and you just set it up.

APIs (Parts 8) — Render, free tier

Create a Web Service on render.com pointed at your API repo; set the start command to node server.js (or uvicorn analytics:app --host 0.0.0.0 --port $PORT for Python). Update the fetch URLs in your front end from localhost to the live Render URL.

9.4 Your Portfolio and the Job Hunt

The Portfolio That Gets Interviews

Three deployed projects beat ten repos. Lead with the Job Finder capstone, the Connected Dashboard, and your strongest Part 1–5 project (weather app or quiz).

Every project gets: a live link, a GitHub link, one screenshot, and three bullet points — what it does, what stack, what problem it solves.

Every README answers in 30 seconds: What is this? What does it look like (screenshot)? How do I run it?

Your portfolio site itself is a portfolio piece: fast, responsive, accessible, Lighthouse 90+.

Positioning Yourself (especially for remote roles)

Title yourself precisely: “Front-End Developer — React, Tailwind, REST APIs”, not “aspiring developer”. You have built and deployed real applications; describe yourself accordingly.

LinkedIn headline and About: lead with what you build and proof: “I build fast, accessible web apps with React. Recent: a job-search platform with live filtering and API integration [link].”

Apply with evidence: every application links your deployed capstone. “Here is something I built that resembles your product” beats any cover letter sentence ever written.

Where to look: LinkedIn, Wellfound, RemoteOK, We Work Remotely, and local tech communities. Contribute answers and small open-source fixes — visible activity compounds.

9.5 Your Daily Professional Habits, From Today

Commit code to GitHub every working day — green squares are a public work ethic.

Read one error message slowly before searching for it.

Rebuild something small from memory weekly (a flexbox navbar, a fetch with error handling, a useState counter).

Run Lighthouse before calling anything finished.

Document as you go — future you is your most important teammate.
